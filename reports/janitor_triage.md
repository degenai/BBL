# Janitor triage

The parking lot for cleanup items that surface during enrichment but aren't worth interrupting a wave for. Each entry has:
- **What** ŌĆö the specific finding
- **Where** ŌĆö path or scope
- **Severity** ŌĆö `block` (gates launch), `warn` (degrades quality but works), `note` (future improvement, not urgent)
- **Surfaced** ŌĆö agent / pass that flagged it
- **Status** ŌĆö `open`, `claimed`, `done`

Whoever runs a janitor pass works from the top down. When an item closes, move it to the closed section at the bottom (or delete; the git log preserves the history).

- **`parse_frontmatter` doesn't strip surrounding quotes from values ŌĆö RESOLVED** ŌĆö Surfaced wave 182 (2026-05-23) when `python researchbot.py --prepare-only --game "Magic: The Gathering"` processed 0 cards. Root cause: `parse_frontmatter` (three identical copies in `researchbot.py`, `bbl_queue.py`, `csv2mdbot.py`) uses `line.partition(":")` and returns the raw value untouched. - **`update_frontmatter_field` YAML-list orphan-items bug** ŌĆö Surfaced wave 181 (2026-05-23) by Alex eyeballing Obsidian on sizzlipede-029 / bisharp-065 / yanma-002. **25 cards corpus-wide** had malformed YAML: `tags_hub: []` (inline empty) immediately followed by block-form `  - item` lines as orphans, which YAML reads as empty-list + invalid-orphans. Source patch landed in `researchbot.update_frontmatter_field`: now imports `BLOCK_LIST_FIELDS` from `bbl_schema` and, when the field being replaced is one of those, the regex captures any contiguous block-form `  - item` lines below the matched line so they get swallowed by the replacement rather than left as orphans. Verified with a synthetic test: a `tags_hub` field with prior block items, re-set with new inline value, leaves zero orphans. Data sweep re-ran in same wave fixed the 88 outstanding cards.
