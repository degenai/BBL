# Janitor triage

The parking lot for cleanup items that surface during enrichment but aren't worth interrupting a wave for. Each entry has:
- **What** — the specific finding
- **Where** — path or scope
- **Severity** — `block` (gates launch), `warn` (degrades quality but works), `note` (future improvement, not urgent)
- **Surfaced** — agent / pass that flagged it
- **Status** — `open`, `claimed`, `done`

Whoever runs a janitor pass works from the top down. When an item closes, move it to the closed section at the bottom (or delete; the git log preserves the history).

- **`parse_frontmatter` doesn't strip surrounding quotes from values — RESOLVED** — Surfaced wave 182 (2026-05-23) when `python researchbot.py --prepare-only --game "Magic: The Gathering"` processed 0 cards. Root cause: `parse_frontmatter` (three identical copies in `researchbot.py`, `bbl_queue.py`, `csv2mdbot.py`) uses `line.partition(":")` and returns the raw value untouched. For values that csv2mdbot wrote via `yaml_safe_scalar` (any value containing `: `, `#`, etc. — including `game: "Magic: The Gathering"` because of the colon), the parsed value retains the literal `"` characters. Filter comparison then fails: `'"magic: the gathering"' != 'magic: the gathering'`. Affected: any `--game` filter for a quote-needing game name (currently just MTG; future colon-containing names like Yu-Gi-Oh would also hit). **Fixed wave 182** in all three copies — added quote-stripping after `.strip()` for both `"..."` and `'...'` wrappers. Status: `done`. Closed 2026-05-23.

- **`update_frontmatter_field` YAML-list orphan-items bug** — Surfaced wave 181 (2026-05-23) by Alex eyeballing Obsidian on sizzlipede-029 / bisharp-065 / yanma-002. **25 cards corpus-wide** had malformed YAML: `tags_hub: []` (inline empty) immediately followed by block-form `  - item` lines as orphans, which YAML reads as empty-list + invalid-orphans. Pattern matched all 25: `^(tags_hub|tags_filter):\s*\[\]\s*\n\s+-\s`. **Fixed inline wave 181** via a one-shot script that strips the orphan `[]` and leaves the block-form items intact. **Root cause:** `researchbot.update_frontmatter_field()` uses regex `^{field}:.*$` matching ONE LINE — when a prep re-init writes `tags_hub: []` to a card that already has block-form items from a prior vision pass, the regex replaces only the first line and the block items below survive as orphans. Triggered by re-prep / `--force` / certain csv2mdbot paths that re-initialize tags fields without checking for existing content. **Code fix applied wave 183 (2026-05-24):** the recurrence proved urgency — between waves 181 and 183, prep runs reintroduced the orphan-items bug on **88 additional cards** corpus-wide. Source patch landed in `researchbot.update_frontmatter_field`: now imports `BLOCK_LIST_FIELDS` from `bbl_schema` and, when the field being replaced is one of those, the regex captures any contiguous block-form `  - item` lines below the matched line so they get swallowed by the replacement rather than left as orphans. Verified with a synthetic test: a `tags_hub` field with prior block items, re-set with new inline value, leaves zero orphans. Data sweep re-ran in same wave fixed the 88 outstanding cards. Status: `done`. Closed 2026-05-24.
